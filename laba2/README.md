## Задание

Модифицировать программу из л/р №1 для параллельной работы по технологии OpenMP. Провести серию экспериментов с разным числом потоков (1, 2, 4, 8) и размерами матриц (200–2000).

- **Входные данные:** файлы `matrix_a`, `matrix_b` с матрицами типа `int`
- **Выходные данные:** время выполнения, ускорение, производительность (GFLOPS), эффективность
- **Верификация:** автоматическая проверка через Python / NumPy

## Реализация

Ключевое изменение по сравнению с л/р №1 — директива OpenMP для распараллеливания внешнего цикла:

```cpp
omp_set_num_threads(num_threads);

#pragma omp parallel for schedule(dynamic, 16)
for (int i = 0; i < n; ++i) {
    for (int j = 0; j < n; ++j) {
        int acc = 0;
        for (int k = 0; k < n; ++k)
            acc += A[i][k] * Bt[j][k];
        C[i][j] = acc;
    }
}
```

Число потоков задаётся аргументом командной строки:
```bash
./build/matrix_mul <threads>
# например: ./build/matrix_mul 4
```

## Анализ результатов

### Ускорение
При 2 потоках ускорение составляет 1.77–1.97× — близко к идеальному ×2. На 4 потоках достигается 2.90–3.82×, эффективность 72–96%. Максимальное ускорение 4.91× зафиксировано на 8 потоках при матрице 2000×2000.

### Производительность
Производительность при 1 потоке стабильна на уровне 0.55–0.57 GFLOPS для матриц до 1600×1600. Падение до 0.25 GFLOPS на 2000×2000 объясняется выходом данных за пределы кэша L3. При 8 потоках максимум — 2.49 GFLOPS (400×400).

### Эффективность
Для 2 потоков эффективность 88–98% — практически линейное масштабирование. Для 4 потоков — 72–95%. Для 8 потоков — 43–61%: ограничено архитектурой 4 физических ядер с Hyper-Threading, которые слабо помогают при вычислительно-интенсивных задачах.
