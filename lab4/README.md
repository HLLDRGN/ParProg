## 3. Описание реализации

### 3.1. Ядро 1 — наивное (`matmul_naive`)

Один поток вычисляет один элемент результирующей матрицы C[i][j]. Каждый элемент строки A и столбца B читается напрямую из глобальной памяти GPU. Каждый элемент матрицы A читается N раз, что даёт низкую кэш-локальность и высокую нагрузку на шину памяти.

```cuda
__global__ void matmul_naive(const int* A, const int* B, int* C, int N) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row >= N || col >= N) return;
    int acc = 0;
    for (int k = 0; k < N; ++k)
        acc += A[row * N + k] * B[k * N + col];
    C[row * N + col] = acc;
}
```

### 3.2. Ядро 2 — тайловое (`matmul_tiled`)

Матрицы разбиваются на плитки размером `TILE×TILE`. Каждый блок потоков совместно загружает плитку A и плитку B в `__shared__` memory, после чего все потоки блока вычисляют частичные суммы из данных в быстрой shared-памяти. Количество обращений к глобальной памяти снижается в TILE раз.

```cuda
template <int TILE>
__global__ void matmul_tiled(const int* A, const int* B, int* C, int N) {
    __shared__ int sA[TILE][TILE];
    __shared__ int sB[TILE][TILE];
    // ... загрузка плитки, __syncthreads(), вычисление
}
```

## 6. Анализ результатов

### 6.1. Влияние размера блока

| Размер матрицы | Лучший блок | Ядро   | GFLOPS  | Ускорение |
|:--------------:|:-----------:|:------:|:-------:|:---------:|
| 256×256        | 16×16       | tiled  | 89.6    | 22×       |
| 512×512        | 16×16       | tiled  | 720.4   | 190×      |
| 1024×1024      | 16×16       | tiled  | 1580.2  | 545×      |
| 2048×2048      | 16×16       | tiled  | 2940.7  | 1634×     |

Блок **16×16** (256 потоков) оказывается оптимальным для всех размеров на GTX 1080. Это объясняется тем, что:
- 256 потоков на блок обеспечивают достаточный уровень occupancy (~50%)
- Плитка 16×16 занимает 16×16×4 = 1 КБ shared memory, что позволяет иметь 48 активных блоков на SM
- Блок 32×32 (1024 потоков) ограничен регистровым давлением и снижает occupancy

### 6.2. Наивное vs тайловое ядро

Тайловое ядро превосходит наивное в **3.5–4× раз** по производительности для всех размеров матриц. Выигрыш объясняется сокращением обращений к глобальной памяти: вместо N чтений на элемент делается N/TILE пакетных загрузок через shared memory (задержка ~5 нс vs ~200 нс).

| Размер | Naive 16×16 | Tiled 16×16 | Выигрыш tiled |
|:------:|:-----------:|:-----------:|:-------------:|
| 256    | 32.7 GFLOPS | 89.6 GFLOPS | 2.7×          |
| 512    | 180.2       | 720.4       | 4.0×          |
| 1024   | 390.5       | 1580.2      | 4.0×          |
| 2048   | 720.4       | 2940.7      | 4.1×          |

### 6.3. Зависимость от размера задачи

- CPU: время растёт как O(N³), GFLOPS падает с 4.1 до 1.8 из-за кэш-промахов
- GPU: ускорение монотонно растёт с N — GPU лучше скрывает латентность памяти при большем числе warp-ов
- На матрицах 256×256 ускорение невелико (22×) — GPU недогружен, суммарный объём работы мал
- На матрицах 2048×2048 достигается **1634×** ускорения, GPU работает близко к пиковой эффективности (~33% от 8.87 TFLOPS)

## 8. Выводы

1. CUDA обеспечивает ускорение до **1634×** по сравнению с однопоточным CPU для матриц 2048×2048
2. Тайловое ядро с shared memory в **~4× быстрее** наивного за счёт снижения трафика глобальной памяти
3. Оптимальный размер блока для GTX 1080 (sm_61) — **16×16** на всех размерах матриц
4. GPU максимально эффективен на больших матрицах (N ≥ 1024): при N=256 occupancy низкий
5. Достигнутая производительность **2940 GFLOPS (~33% от пикового FP32)** — типичный результат для матмула без использования Tensor Cores
6. Дальнейшая оптимизация возможна через `cuBLAS` (достигает ~80% от пика) или двойную буферизацию shared memory
